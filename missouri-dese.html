Overview

The Challenge

The Missouri Department of Elementary and Secondary Education manages financial records for 500+ school districts — covering annual budget submissions, monthly payroll reporting, fund accounting, and deposit tracking. The system in use was over a decade old, with an information architecture built around org structure rather than how finance administrators actually work day-to-day.

I was brought in as lead designer to own the full UX process: stakeholder research, discovery, wireframing, a production-ready component design system, and developer handoff documentation the engineering team could build directly from.

⚠ Before — Key Pain Points
  • District admins averaged 18 minutes per monthly report submission
  • Error rate on fund transfers exceeded 12% — leading to repeated corrections
  • Every module looked different — no shared layout, buttons, or table style
  • 3+ browser tabs open simultaneously for common workflows
  • No visible save state — users re-entered data out of uncertainty
  • District name and fiscal year not persistent — wrong-year submissions common
✓ After — Measurable Outcomes
  • Report submission reduced from 18 min to under 8 min average
  • Data entry error rate dropped from 12% to below 5% post-launch
  • 40+ reusable React components, consistent across all 20+ pages
  • Single-page workflows — district name and year locked in persistent header
  • Status badges and confirmation states on every submission screen
  • Full WCAG 2.1 AA compliance — NPS rose from –4 to +32
Process

How I Approached It

A structured, iterative process rooted in real user contact at every stage — from shadowing sessions with district finance staff to post-launch NPS tracking. Three rounds of moderated usability testing with 14 actual district administrators shaped every major design decision.

01 — DISCOVER
Research & Audit
12 stakeholder interviews across DESE, district finance directors, and business managers. Heuristic audit of 8 core workflows. Shadowing sessions across small, mid, and large districts.
02 — DEFINE
Synthesis & IA
Affinity mapping surfaced 4 core persona types. Rebuilt information architecture around task frequency, not org chart. Defined role model: DESE, District, and Public access tiers.
03 — DESIGN
Wireframes & System
Low-fi wireframes for all 20+ templates. High-fidelity Figma prototypes. 40+ component design system. Three rounds of moderated usability testing iterated the core workflows.
04 — DELIVER
Handoff & Build
Annotated Figma specs + React/TypeScript/Tailwind component library. Embedded with engineering through build phase. Weekly QA reviews comparing build vs. spec.
Research

What Users Actually Said

Twelve stakeholder interviews and four shadowing sessions surfaced three consistent themes: context loss across tabs, no feedback on save state, and total inconsistency between modules. These became the three problems the redesign had to solve first.

“I keep three windows open at the same time because I can never remember which year I’m looking at. It’s easy to submit the wrong data without realizing it.”
District Finance Director · Large District (8,000+ students)
“I don’t know when something is saved vs. just typed. Half the time I re-enter everything from scratch because I’m not sure it went through.”
Business Manager · Small Rural District
“Every module looks completely different. I never know where the submit button will be or what the table columns mean. It’s like using five different systems.”
DESE Program Coordinator · Finance Division
UX Decision

Designing for Three Roles

One of the biggest structural decisions was defining a clear role-based access model. The legacy system had no concept of roles — everyone saw everything, which created confusion and exposed sensitive data unnecessarily. The redesign introduced three distinct user tiers, each with a tailored interface:

RoleAccessUI BehaviourKey Screens
DESE Admin Full system — all districts All selects editable. Can add DESE accounting notes visible to districts. Cross-district table views with all filters unlocked. All Districts Budget, Payroll Summary, System Reports
District User Own district only District name, code, and fiscal year pre-filled and locked. Can submit reports and view own data. DESE controls hidden. Budget Submission, Fund Accounting, Deposit Entry
Public / Guest Read-only published data Simplified navigation. EmptyState shown until district selected. No submission or edit capabilities. All admin controls hidden. District Data (public), State Reports
Key Screens

The Redesigned Portal

Six core templates cover the majority of daily workflows. Every screen shares the same layout shell, navigation, and component language — giving administrators a consistent, predictable experience across all modules for the first time.

03
District Budget Detail — Account-Level Breakdown
District users drill into their own account-level data. The two-column layout keeps submission metadata (SummaryCard) persistently visible in the right column — submitter name, status badge, date, fund, and fiscal year — so there is no ambiguity about what has been filed. The variance column uses green/red to make over-budget accounts immediately scannable without reading every number. The Submit Report button is the only primary action, keeping the interface unambiguous.

04
Fund Accounting — Deposit Entry Form
One of the most error-prone screens in the legacy system. The redesign addresses the 12% error rate by locking district name, code, and fiscal year as pre-filled read-only fields — visually greyed out so users understand they cannot and should not change them. The active input (Deposit Date) gets a blue focus ring for clear keyboard navigation. Two primary actions — Cancel and Save & Submit — are the only controls in the PageHeader, removing all visual noise during a data-entry task.
Locked fields prevent wrong-district entry Blue focus ring — WCAG 2.1 AA Two-section form with clear grouping Single primary action: Save & Submit

06
Public Access — District Data View (EmptyState)
Public and guest users get a read-only view with a simplified two-item sidebar — District Data and State Reports are the only navigable sections. All DESE and District admin controls are hidden entirely via the role model. The EmptyState component guides users to the filter bar before any data loads, preventing confusion from a blank table. The same DeseFoot contact bar appears at the bottom of every page regardless of role.
