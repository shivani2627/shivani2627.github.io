Overview

DA offices running on spreadsheets and paper

When I joined the PROTECT project, Wisconsin District Attorney offices were tracking active criminal cases across a combination of spreadsheets, shared drives, and paper files. No single source of truth, no audit trail, and no way to understand case workload at a glance.

The State of Wisconsin's Department of Administration needed a purpose-built case management system. I was brought in as the sole UI/UX designer and embedded with the team for 18 months — long enough to understand the domain deeply, build trust with stakeholders, and deliver five distinct design efforts under the PROTECT umbrella.

My primary challenge: desktop and web users had fundamentally different interaction patterns. DAs reviewing case files needed dense, information-rich desktop interfaces. Supervisors and clerks accessing the web portal needed streamlined task flows. One IA would not serve both. Solving that split became the structural foundation for every design decision that followed.

⚠️  Before PROTECT
  • Cases tracked in Excel — no audit trail, version conflicts
  • No visibility into DA workload across the office
  • eFiling done via email attachments — error-prone, manual
  • Desktop and web users forced into the same interface
  • No brand or visual identity — felt like a temporary system
  • XCQ queries ran in a separate legacy CLI tool
✦  After PROTECT
  • Single platform with full case lifecycle tracking and audit log
  • Dashboard-driven workload visibility for supervisors
  • Dedicated eFiling portal with validation and status tracking
  • Separate IA for desktop (dense/keyboard) vs. web (task-focused)
  • Full brand identity: logo, colour system, typography, icon set
  • XCQ querying rebuilt as a first-class UI module for all staff

Discovery & Research

Map the lifecycle before touching the UI

Before sketching a single screen, I spent the first six weeks running discovery workshops with District Attorneys, investigators, administrative clerks, and IT stakeholders. The goal was to understand the full case lifecycle — not just the task flows, but the mental model each user type carried into their work.

The most important finding came from observing DAs vs. clerks side by side: DAs think in cases (deep, contextual, non-linear) while clerks think in tasks (linear, batch-oriented, time-sensitive). The same data set, accessed with completely different intent.

This observation became the structural principle the entire IA was built on.

"When I'm reviewing a case I need everything — charges, history, filings, related parties — all visible without switching screens. I can't afford to hunt for information when I'm in court prep."
— District Attorney, Milwaukee County (research session, March 2024)
"I process 40–60 filings a day. I just need to know what's new, what needs action, and what's done. I don't need to see the whole case file every time."
— Administrative Clerk, Dane County (contextual inquiry, April 2024)
"Right now I run XCQ queries in a terminal. Nobody else in the office knows how. That knowledge shouldn't live in one person's head."
— Senior Investigator (stakeholder workshop, March 2024)

The Five Workstreams

One project, five distinct design efforts

PROTECT wasn't a single app — it was a platform ecosystem. Each workstream had its own users, constraints, and design challenges. Here's what I delivered across each:

01
Brand Identity

Created the PROTECT visual identity from scratch: logo (shield + document mark), purple–black colour palette, DM Sans typography, and an icon set that matched the platform tone — authoritative but approachable. The brand had to feel like a real government system, not a prototype.

Logo designGIMPBrand guidelinesIcon setColour system
02
Desktop App Revamp

Complete redesign of the primary DA desktop application. Dense sidebar navigation, full-width case view with linked entities, keyboard-first interaction patterns, and collapsible panel system for multi-case workflows. The key constraint: DAs cannot afford even a single extra click during court preparation — every path had to be muscle-memory fast.

Desktop IADense data UIKeyboard navFigmaComponent library
03
Details Pane Web

Responsive web companion for clerks and supervisors. Task-oriented flows, batch action patterns, and streamlined case detail view optimised for high-volume daily use. Where the desktop app was built for depth, this was built for throughput — clerks processing 40–60 filings a day needed zero friction.

Responsive webTask flowsBatch actionsSupervisor views
04
XCQ Web Module

Rebuilt a legacy command-line query tool as a first-class UI. Visual query builder with field selectors, result preview, export options, and saved query patterns — making advanced case search accessible to all staff, not just the one investigator who knew the terminal commands. Democratising access to institutional knowledge.

Query builder UIResults displayExport patternsSaved queries
05
eFiling Portal

End-to-end eFiling experience replacing email-attachment workflows. Stepped submission flow with real-time validation, document preview, filing status tracking, and confirmation notifications. The goal: eliminate the "did my email go through?" ambiguity that was causing clerks to follow up every filing with a phone call. We eliminated those calls entirely.

Reflection

What I'd do differently

01
Validate the IA split earlier
The desktop vs. web IA distinction emerged from research in week 4, but I could have prototyped and validated it with users in week 2. Earlier structural validation saves mid-project rework.
02
Document design rationale in real-time
18 months of decisions accumulate fast. I should have kept a running decision log from day one — referencing "why we made this call" in stakeholder reviews would have reduced revisiting settled ground.
03
Push for a design system earlier
The component library came together in month 6. If I'd established it in month 2, the later workstreams would have shipped faster. Solo design on a long engagement demands a system-first mindset.
