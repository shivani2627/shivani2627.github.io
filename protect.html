Overview

DA offices running on spreadsheets and paper

When I joined PROTECT, Wisconsin District Attorney offices were tracking active criminal cases across spreadsheets, shared drives, and paper files. No single source of truth, no audit trail, no way to understand case workload at a glance.

I was brought in as the sole UI/UX designer and embedded with the team for 18 months — long enough to understand the domain deeply, build trust with DA stakeholders, and deliver five distinct design efforts under the PROTECT umbrella.

The core structural challenge: desktop and web users had fundamentally different interaction patterns. DAs reviewing case files needed dense, information-rich desktop interfaces. Clerks and supervisors accessing the web portal needed streamlined task flows. One IA would not serve both — solving that split became the foundation for every design decision that followed.

⚠  Before PROTECT
  • Cases tracked in Excel — no audit trail, version conflicts
  • No visibility into DA workload across the office
  • eFiling done via email attachments — error-prone, manual
  • Desktop and web users forced into the same interface
  • No brand or visual identity — felt like a temporary system
  • XCQ queries ran in a separate legacy CLI tool only one person knew
✦  After PROTECT
  • Single platform with full case lifecycle tracking and audit log
  • Dashboard-driven workload visibility for supervisors
  • Dedicated eFiling portal with validation and status tracking
  • Separate IA for desktop (dense/keyboard) vs. web (task-focused)
  • Full brand identity: logo, colour system, typography, icon set
  • XCQ rebuilt as a first-class visual UI module accessible to all staff

Discovery

Map the lifecycle before touching the UI

Before sketching a single screen, I spent the first six weeks running workshops with District Attorneys, investigators, administrative clerks, and IT stakeholders. The goal was to understand the full case lifecycle — not just task flows, but the mental model each user type carried into their work.

The most important finding: DAs think in cases (deep, contextual, non-linear) while clerks think in tasks (linear, batch-oriented, time-sensitive). Same data set, completely different intent. This became the structural principle the entire IA was built on.

"When I'm reviewing a case I need everything — charges, history, filings, related parties — all visible without switching screens. I can't afford to hunt for information when I'm in court prep."
— District Attorney, Milwaukee County (research session, March 2024)
"I process 40–60 filings a day. I just need to know what's new, what needs action, and what's done. I don't need to see the whole case file every time."
— Administrative Clerk, Dane County (contextual inquiry, April 2024)
"Right now I run XCQ queries in a terminal. Nobody else in the office knows how. That knowledge shouldn't live in one person's head."
— Senior Investigator (stakeholder workshop, March 2024)
Design Process

Discovery through handoff

01
Stakeholder workshops
6 weeks of facilitated sessions mapping case lifecycle, user types, and system constraints across DA offices statewide.
02
IA & flows
Separate IA for desktop vs. web. FigJam journey maps validated the structural split before any visual design began.
03
Iterative design
Lo-fi → mid-fi → hi-fi across all 5 workstreams. Review cycles with PMs, DAs, and dev every 2 weeks throughout.
04
Handoff & UAT
Annotated Figma specs and component library. Facilitated UAT sessions with actual DA staff to validate before build.

Reflection

What I'd do differently

01
Validate the IA split earlier
The desktop vs. web distinction emerged in week 4. Prototyping and validating it with users in week 2 would have saved mid-project rework.
02
Document decisions in real-time
18 months of decisions accumulate fast. A running decision log from day one would have reduced revisiting settled ground in stakeholder reviews.
03
Build the design system earlier
The component library came together in month 6. If I'd established it in month 2, later workstreams would have shipped faster. Solo long-engagement = system-first mindset.
